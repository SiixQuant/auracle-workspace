# SUMMARY
The engine (Houston, /Users/siix/Downloads/auracle) exposes ~130 routes across three auth domains: a cookie+CSRF /ui/api/* JSON surface (the IDE/launcher backbone, wired in auracle/houston/web/router.py from domain modules in web/views/), a bare X-API-Key /api/* REST surface (routes/*.py), and an unauthenticated /auth/* keyless device flow (routes/auth_device.py) that mints Ed25519-signed entitlement JWTs. Clients authenticate to /ui/api via the auracle_session cookie (which literally holds the per-user API key) plus a double-submit auracle_csrf token on mutations; the launcher bootstraps this by exchanging the on-box handoff secret (<install>/data/keys/.ide-handoff-secret) for the owner key at POST /ui/api/ide/provision-local, then writes {engine_url, api_key} to ~/.config/auracle/auracle.json for the IDE. The launcher's device-flow sign-in (auth_device.rs) currently targets the LOCAL engine only (ENGINE_BASE=http://127.0.0.1:1969) — no HQ-first topology exists in code — storing signed_jwt/refresh_token in the keychain with no expiry validation or refresh yet. IDE install/update (ide_update.rs) is entirely GitHub-release-driven: newest auracle-v* tag on SiixQuant/auracle-ide, .dmg asset + mandatory SHA-256 sidecar (fail-closed), hdiutil mount → atomic swap into "/Applications/Auracle IDE.app", with ~/.auracle/ide_version as the sole version truth — a nimbalyst-based IDE must reproduce exactly this release shape to slot in.

## Engine auth mechanisms (3 credential planes)
(1) Cookie plane for /ui/*: auracle_session cookie HOLDS the user's per-user API key directly (no session table) — auracle/houston/web/views/_common.py current_user/require_user (lines 122-135); minted by POST /ui/login (web/views/auth.py:93), first-run POST /ui/setup (auth.py:400, gated by one-shot setup token: auracle/houston/setup_token.py, file /data/keys/.setup-token, via ?token= or auracle_setup_token cookie, opt-out AURACLE_DISABLE_SETUP_TOKEN), and GET /ui/ide-login?nonce= (web/views/connect_agent.py:465) consuming a 60s single-use nonce. (2) Header plane: X-API-Key (auracle/houston/auth.py) accepting HOUSTON_ADMIN_KEY→role owner, HOUSTON_SERVICE_KEY→role service, or per-user keys; role hierarchy viewer<trader<user=service<admin/owner (ROLE_RANK line 140); require_admin/require_role gates; auracle/houston/license_gate.py adds require_valid_license + require_tier for tier-locked surfaces. (3) On-box handoff plane: X-Auracle-Handoff-Token header proving possession of <install>/data/keys/.ide-handoff-secret (auracle/houston/ide_handoff.py, 0600 file). Some routes accept cookie OR X-API-Key (_require_user_or_api_key in connect_agent.py:214, used by GET /ui/api/ide/connect-check).

## CSRF contract a client must speak
auracle/houston/csrf.py: double-submit cookie. GET to any /ui path issues auracle_csrf cookie (HttpOnly=false, SameSite=Lax, 30d). Every POST/PUT/DELETE/PATCH under /ui must echo it via X-CSRF-Token header (or csrf_token form field) matching the cookie, else 403. Exempt prefixes: /webhooks/, /healthz, /api/, /auth/, /admin/, /legal/, /license/, /checkout/, /portal/. Exempt /ui paths: /ui/login, /ui/setup, /ui/setup/license, /ui/api/ide/provision-local, /ui/api/ide/session-handoff. IMPORTANT: the root-mounted live-deploy routes (/deploy/live, /deployments/*) are NOT under any exempt prefix, so their POSTs require cookie + CSRF too. The launcher's canonical pattern (auracle-desktop/src-tauri/src/commands/engine_auth.rs fetch_csrf, line 145): GET /ui/api/status with X-API-Key + 'Cookie: auracle_session=<key>', harvest Set-Cookie auracle_csrf, replay it as both cookie and X-CSRF-Token on the mutation.

# RISKS
- Mutating /ui/api/* and root-mounted /deploy/live + /deployments/* endpoints require the auracle_session COOKIE plus double-submit CSRF (X-CSRF-Token matching the auracle_csrf cookie) — a new IDE client that sends only X-API-Key will 401/403 on most of the surface; only a few routes (e.g. /ui/api/ide/connect-check) accept the header. Any nimbalyst client must reproduce the fetch_csrf dance in auracle-desktop/src-tauri/src/commands/engine_auth.rs.
- Auth is inconsistent across the surface: cookie-only (most /ui/api + /deployments), X-API-Key-only (bare /api/*, /auth/register), owner-only vs user-role gates, and loopback-only secret egress (/ui/api/settings/ai-key). A client SDK needs a per-endpoint auth map, not one scheme (the '3 transport bypasses to consolidate' noted in the 2026-07-01 settings-remodel memory).
- The launcher's keyless sign-in targets the LOCAL engine (ENGINE_BASE=127.0.0.1:1969), not HQ — the planned HQ-first + local-fallback topology from the production roadmap is NOT in code yet, and /auth/device/start requires the engine's Resend/SMTP email to be configured (memory: 'Resend coded but unconfigured'), so sign-in silently sends nothing on a default install.
- No refresh/expiry handling in the launcher: sign_in_status only checks keychain presence of the refresh token; /auth/refresh and /auth/device/poll are implemented server-side but never called — a stale/expired JWT still reads as 'signed in'.
- IDE update path is macOS-aarch64-only and fail-closed on the SHA-256 sidecar: any nimbalyst release missing the bare-hex Auracle-aarch64.dmg.sha256 asset (or the auracle-v* tag convention, or a readable CFBundleShortVersionString) will be refused by every installed launcher.
- The live-deploy router docstring (auracle/houston/routes/live.py line 15) claims the routes sit under /ui/api externally, but they are mounted at root with no Caddy rewrite — clients relying on /ui/api/deployments/* would 404; the real paths are /deployments/*.
- The auracle_session cookie stores the raw per-user API key (no server-side session), so any client persisting cookies is persisting a live long-lived credential; treat it with keychain-grade care in the new IDE.
