# SUMMARY
Nimbalyst has a mature, VS Code-grade extension system: manifest-driven contributions (custom editors per glob, panels in 4 placements, settings panels, commands/keybindings, AI chat slash commands, AI tools, agent providers, themes, Claude Code plugins) declared in manifest.json, React components loaded into the renderer via blob-URL ES modules with host-provided externals (packages/runtime/src/extensions/ExtensionLoader.ts, ~2,200 lines). The EditorHost contract (packages/extension-sdk/src/types/editor.ts) is the single editor interface used by both built-ins (MonacoEditor/MarkdownEditor in packages/runtime/src/editors) and extensions, providing load/save/dirty/diff/source-mode/collab/AI-context/menu APIs. Extensions ship two ways: pre-bundled first-party (packages/extensions/* built by packages/electron/build/build-extensions.js into resources/extensions, treated as the app's own trust domain with consent auto-granted) or marketplace .nimext zips downloaded from a Cloudflare Worker + R2 CDN with SHA-256 verification (packages/electron/src/main/ipc/ExtensionMarketplaceHandlers.ts). Sandboxing is honest but thin: renderer extension code runs unsandboxed in the app renderer (free fetch/DOM), while native-code "backend modules" run in utility processes gated by an allowlist + explicit user consent (packages/extension-sdk/src/types/permissions.ts). Verdict for Auracle: features (a) Connections surface, (b) live-trading dashboard, (c) deploy wizard, and (d) node-graph canvas are all cleanly buildable as pre-bundled extensions with existing precedents in-repo; (e) a hard startup sign-in gate and (f) swapping the default agent persona need small core hooks, though ~80% of (f) (new provider + slash commands + skills) is already extension-shaped via aiAgentProviders + claudePlugin/agentWorkflows contributions.

## 3. Loading, installation, sandboxing, pre-bundling
Discovery/loading: platform-agnostic ExtensionLoader (packages/runtime/src/extensions/ExtensionLoader.ts) validates manifest.json then loads the bundle via ExtensionPlatformServiceImpl.loadModule (packages/electron/src/renderer/services/ExtensionPlatformServiceImpl.ts lines 274-320): reads the JS, wraps it in a Blob URL, imports with es-module-shims + an import map so react/lexical/yjs/@nimbalyst/runtime resolve to the HOST's copies (REQUIRED_EXTERNALS in packages/extension-sdk/src/externals.ts). Three install sources: (1) BUILT-IN: packages/extensions/* in dev, resources/extensions in packaged app (packages/electron/src/main/extensions/builtinExtensionsDirectory.ts; built by packages/electron/build/build-extensions.js which runs each extension's build before electron-builder) — 'same trust domain as the app itself', backend-module allowlist admits unconditionally and consent prompt is SKIPPED (isBuiltinExtensionPath, lines 49-61), so YES, pre-bundled first-party extensions are a fully supported, privileged path; (2) MARKETPLACE: registry.json + .nimext zips served by a Cloudflare Worker over R2 (packages/marketplace/README.md, src/index.ts; curated set in packages/marketplace/release-extensions.txt), installed by packages/electron/src/main/ipc/ExtensionMarketplaceHandlers.ts (download → SHA-256 verify → extract-zip to <userData>/extensions → validate → atomic swap; also installFromGitHub release-asset path); (3) DEV MODE via packages/electron/src/main/services/ExtensionDevService.ts + the extension-dev-kit extension. Sandboxing (packages/extension-sdk/src/types/permissions.ts + packages/extension-sdk-docs/permissions.md): renderer extension code is NOT sandboxed — it runs in the app renderer with DOM + fetch; the legacy filesystem/ai/network booleans gate host IPC surfaces (exec, AI service) but not ambient browser APIs. Real enforcement exists only for host-brokered catalog permissions (workspace-files, nimbalyst-database-read/write, secrets-read, mcp-server-register) and for backendModules: native code runs in utilityProcess.fork/worker_threads, default-disabled, first-use consent prompt ('run native code with the same trust as your user account'), marketplace extensions need an entry in packages/electron/src/main/extensions/backendModuleAllowlist.ts, dev installs need NIMBALYST_ALLOW_DEV_BACKEND_MODULES=1.

## 5e. VERDICT — 'Sign in with Auracle' startup gate
CORE FORK REQUIRED for a real gate. There is no contribution point that runs before app UI or can block workspace access: extensions are discovered and loaded after the renderer shell boots (ExtensionLoader in the app tree), hostComponents mount inside the running app (packages/electron/src/renderer/components/ExtensionHostComponents.tsx renders them as ordinary children), and a user can disable any extension from Settings — so an extension 'gate' is a curtain, not a lock. A floating panel/hostComponent overlay with activationEvents:['onStartup'] (panel.ts line 87) can APPROXIMATE a soft sign-in prompt, but enforcement (blocking file/agent access until authed) needs a small core hook: either a first-party patch to the app entry (packages/electron/src/renderer, ~one gating component) or a new host-owned 'authGate' contribution honored before workspace mount. Since you'd ship a fork/whitelabel anyway, this is a contained core change, not an architectural fight.

# RISKS
- API stability: host.data.query() is explicitly documented as unstable ('intended for built-in extensions today... will be redesigned when storage ports to native SQLite' — panel.ts lines 361-366), and several surfaces carry deprecated aliases; pin to a tested host version if building against the SDK rather than forking.
- The permission model is consent-theater beyond the brokered catalog: renderer extensions get unrestricted fetch/DOM regardless of permissions.network, and a granted backend module has full native access — fine for first-party pre-bundled code, but it means Auracle cannot rely on the extension sandbox to contain third-party marketplace extensions handling trading credentials.
- Built-in extensions (git) reach past the SDK into private window.electronAPI IPC channels; if Auracle copies that pattern, its extensions silently depend on host internals and lose the 'no fork' property the SDK promises.
- The startup-gate (5e) and default-agent-swap (5f) core patches are small but live in upstream-hot files (renderer app entry, AgentProviderRegistry/session creation); if you track upstream Nimbalyst, these are the two standing merge-conflict points to isolate behind minimal seams.
- Marketplace distribution assumes Nimbalyst's Cloudflare Worker/R2 infra (extensions.nimbalyst.com hardcoded in registry generation — packages/marketplace/scripts/generate-registry.sh); shipping Auracle extensions outside the app bundle would require either the pre-bundled path (recommended) or standing up a parallel registry.
- Backend modules for marketplace installs require an entry in backendModuleAllowlist.ts baked into the app — any Auracle extension needing native code effectively must ship pre-bundled or in a forked build anyway.
