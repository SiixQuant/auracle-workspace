# SUMMARY
The Auracle IDE fork at /Users/siix/Downloads/auracle-ide (checked out on branch feat/live-deploy-panel) carries 15 owner-added crates on the current lineage: 8 auracle_* crates plus 7 finance dock-panel crates (blotter_panel, incidents_panel, runs_dock, runway_rail, schedules_panel, strategies_panel, validation_rail); everything else under crates/ (including sidebar, csv_preview, which_key, mermaid_render) is upstream Zed. All finance features talk to the local engine over one loopback HTTP contract — base http://127.0.0.1:1969, auth = X-API-Key header + auracle_session cookie, mutations add a double-submit CSRF token harvested from GET /ui/api/status, config in <config-dir>/auracle.json with AURACLE_ENGINE_URL/AURACLE_API_KEY env overrides. The architecture is consistently split into gpui-free "reducer" crates/functions (auracle_live entirely, plus pure helpers in auracle_connections) that port to TypeScript nearly verbatim, and thin GPUI render/poll shells that must be rebuilt in React. Critically, a second, richer finance lineage (Studio cockpit, backtest-results tab, QuantConnect import, Flow node canvas, Desk overview, spec compiler — 36 auracle_* crates) exists ONLY on divergent unmerged branches (feat/flow-shell, studio, qc-import-divergence, auracle-*-core); it is absent from main and the working tree, so a port scoped to the current checkout would silently drop Studio, Flow, and QC import. The auracle_live/auracle_live_ui deploy crates are likewise only on the current feature branch, not yet on main.

# RISKS
- Branch divergence is the biggest porting hazard: Studio (cockpit/backtest-results/spec-compiler), Flow (node canvas), QC import, Desk overview, and the auracle_actions/artifact/session core exist ONLY on unmerged branches (feat/flow-shell, studio, qc-import-divergence, auracle-*-core) that have DIVERGED from main — porting from the working tree or main alone silently drops them, and the two lineages contain conflicting duplicates (auracle_status vs auracle_status_view; schedules_panel inline logic vs auracle_schedules reducer; auracle_connect_state vs inline ConnectModal logic).
- auracle_live + auracle_live_ui (Deploy wizard, Live Algorithms, orders ledger) live only on the current feat/live-deploy-panel branch — not on main — so 'main' is not a complete finance-surface source either.
- The API contract is not uniform: /backtest and /backtest/studio (flow-shell lineage) are NOT under the /ui/api prefix; some GETs send only the auracle_session cookie without X-API-Key (auracle_live_ui::engine_get, auracle_status::poll_once); the TS host should standardize on both headers plus the double-submit CSRF dance for every mutation, or older/newer engines will intermittently 401.
- Per-row order cancel in the Blotter is blocked on an engine contract gap (feed lacks broker_order_id; cancel route addresses the broker book, not DB row ids) — porting the UI without carrying that constraint would create a dishonest control the codebase deliberately avoids.
- POST /ui/api/settings/ai-key returns a PLAINTEXT provider key and is safe only because transport is loopback; if the TS host ever talks to a non-local engine this endpoint (and key-in-body PUT /ui/api/settings) becomes a secret-leak path.
- Deploy wizard broker list is hardcoded (clearstreet/alpaca/ibkr in crates/auracle_live_ui/src/deploy_wizard.rs) rather than driven by GET /ui/api/connections?kind=broker — replicate the endpoint-driven list, not the constant.
- The engine's honesty invariants (connected only from engine state, no green during in-flight tests, actions offered only where the lifecycle allows, never fabricate values) are enforced in scattered pure functions and comments, not in the API — a React rewrite can easily regress them; port the reducer unit tests (auracle_live, auracle_connections, and the flow-shell *_state crates) as the spec.
