# SUMMARY
Nimbalyst's agent layer launches Claude Code through the @anthropic-ai/claude-agent-sdk query() API (not by scripting the interactive CLI), with the adapter stack in packages/runtime/src/ai/server/providers/ (ClaudeCodeProvider.ts + claudeCode/sdkOptionsBuilder.ts are the core), while Codex runs via spawned `codex exec`/`codex app-server` subprocesses behind protocol classes; a subscription-billed genuine-CLI provider exists but is unimplemented scaffolding. A quant-domain agent is highly preconfigurable through documented seams: a prompt addendum (or full replacement via the meta-agent path) in runtime/src/ai/prompt.ts, SDK-native CLAUDE.md/.claude/rules loading, .claude/commands + .claude/skills discovery (SlashCommandService/AgentWorkflowService), and extension-shipped Claude plugins + MCP servers injected into query() options. Tasks are a tracker system whose type schemas are YAML files in {workspace}/.nimbalyst/trackers/ but whose items are database-first in PGLite (with optional E2E-encrypted team sync), so the README's "plain markdown files" framing is partially historical. All session/workspace state lives in a local PGLite/SQLite database under ~/Library/Application Support/@nimbalyst/electron/, with Claude's own JSONL sessions imported from ~/.claude/projects/. The mobile app syncs E2E-encrypted sessions through wss://sync.nimbalyst.com using the open wire protocol in packages/collab-protocol, but the server itself is a closed separate project and the production URL is hardcoded outside dev mode; sync is fully optional and off by default. For auth, a complete Stytch B2B stack (Google OAuth + magic link via deep links, safeStorage persistence, org/team JWTs) already exists but never gates startup — the UnifiedOnboarding first-run modal and WorkspaceManager window are the obvious places to insert a mandatory sign-in.

# FINDINGS

## 1a. How Claude Code sessions are launched: Agent SDK, not raw CLI spawn
Primary path is the Claude Agent SDK: `packages/runtime/src/ai/server/providers/ClaudeCodeProvider.ts` (3,637 lines) imports `query()` from `@anthropic-ai/claude-agent-sdk` (^0.3.198, pinned in packages/electron/package.json and packages/runtime/package.json; the SDK's native binary + vendored ripgrep are unpacked from asar per packages/electron/package.json asarUnpack rules). The full query() options object (systemPrompt, settingSources, mcpServers, permissionMode, canUseTool, hooks, env, resume/forkSession, plugins) is assembled in `packages/runtime/src/ai/server/providers/claudeCode/sdkOptionsBuilder.ts`; the binary path is resolved by `claudeCode/cliPathResolver.ts` with a user-overridable custom path. Session resumption maps Nimbalyst sessionId -> Claude session id (`options.resume`), branching uses `options.forkSession`. A second provider id `claude-code-cli` (`packages/runtime/src/ai/server/providers/ClaudeCodeCliProvider.ts`) is reserved for driving the genuine `claude` CLI on a Pro/Max subscription inside the embedded ghostty terminal — its sendMessage still throws 'Phase 1 not implemented'; renderer plumbing exists at packages/electron/src/renderer/components/UnifiedAI/claudeCliInputRouting.ts and store/listeners/claudeCliTerminalListeners.ts.

## 1b. Provider/adapter layer layout (all agents)
Adapter layer lives in `packages/runtime/src/ai/server/`: `providers/BaseAgentProvider.ts` (shared base), `ProviderFactory.ts`, `SessionManager.ts`, `providers/ProviderSessionManager.ts`. Codex: `providers/OpenAICodexProvider.ts` with two transports — `protocols/CodexSDKProtocol.ts` (spawns `codex exec --experimental-json` via @openai/codex-sdk) and `protocols/CodexAppServerProtocol.ts` (spawns `codex app-server --listen stdio://`, JSON-RPC v2, the default end state); plus `OpenAICodexACPProvider.ts` (ACP), `OpenCodeProvider.ts`, `CopilotCLIProvider.ts`, `ClaudeProvider.ts`/`OpenAIProvider.ts`/`LMStudioProvider.ts` (plain chat), `ExtensionAgentProvider.ts` (extension-defined agents). CLI detection/install (which binary, PATH repair for Dock-launched Electron, Windows .cmd handling) is `packages/electron/src/main/services/CLIManager.ts`; IPC surface is `packages/electron/src/main/ipc/ClaudeCodeHandlers.ts`, `ClaudeCodeSessionHandlers.ts`, `SessionHandlers.ts`. Existing CLI-created sessions are imported by scanning `~/.claude/projects/**/*.jsonl` (`packages/electron/src/main/services/ClaudeCodeSessionScanner.ts` + `ClaudeCodeSessionSync.ts`). Transcripts are two-tier: raw `ai_agent_messages` -> canonical `ai_transcript_events` via TranscriptTransformer (docs/TRANSCRIPT_ARCHITECTURE.md, parsers in runtime/src/ai/server/transcript/).

## 1c. Preconfiguring a quant domain agent: four documented injection points
docs/SYSTEM_PROMPT_CUSTOMIZATION.md is the authoritative map. (1) System prompt: `buildClaudeCodeSystemPrompt()` in `packages/runtime/src/ai/prompt.ts` is appended to the SDK's claude_code preset via `systemPrompt:{type:'preset',preset:'claude_code',append}` (sdkOptionsBuilder.ts:204-210); the meta-agent path (`buildMetaAgentSystemPrompt`, `isMetaAgent` flag) FULLY replaces the preset — an existing seam for a domain persona. (2) CLAUDE.md: loaded natively by the SDK from ~/.claude/CLAUDE.md, {workspace}/CLAUDE.md and .claude/rules/*.md, gated by `settingSources` (user/project/local toggles from ClaudeCodeSettings, sdkOptionsBuilder.ts:178-194) — so a quant workspace template just needs a CLAUDE.md + .claude/ tree on disk. (3) Skills/plugins: extensions declare `contributions.claudePlugin` in manifest.json; `ClaudeCodeDeps.extensionPluginsLoader` (runtime/src/ai/server/providers/claudeCode/dependencyInjection.ts:119, fed by getNativeClaudePluginPaths in packages/electron/src/main/ipc/ExtensionHandlers.ts:732) injects them as `options.plugins` into query() — a 'Quant Strategy' extension could ship skills, slash commands and MCP tools this way, installable from the marketplace (packages/marketplace). (4) MCP: `McpConfigService.getMcpServersConfig()` (runtime/src/ai/server/services/McpConfigService.ts) + internal servers in packages/electron/src/main/mcp/httpServer.ts — a QuantConnect/backtest MCP server slots in here. Permission presets can be pre-written via `packages/electron/src/main/services/ClaudeSettingsManager.ts` which reads/writes .claude/settings.json / settings.local.json / ~/.claude/settings.json. Caveat: CLAUDE.md forbids env-var API-key fallbacks (billing incident, root CLAUDE.md:29-39) and sdkOptionsBuilder strips ANTHROPIC_API_KEY/OPENAI_API_KEY from all env sources.

## 2a. Custom slash commands + skills discovery
`packages/electron/src/main/services/SlashCommandService.ts` scans, for Claude providers: {workspace}/.claude/commands/**.md (recursive, markdown with frontmatter parsed by `CommandFileParser.ts`), {workspace}/.claude/skills/<name>/SKILL.md, ~/.claude/commands, ~/.claude/skills, and ~/.claude/plugins (plugin-bundled skills); for Codex: {workspace}/.agents/skills. Results merge with SDK-reported commands/skills and a per-provider builtin catalog (5s cache). `packages/electron/src/main/services/AgentWorkflowService.ts` (1,063 lines) is the unified 'agent workflow' registry over workspace claude commands/skills, extension workflows, legacy plugins and provider-native commands, with source toggles (`agentWorkflowSources` in src/main/utils/store.ts:126-131) and cross-provider export (`agentWorkflowExports.codexEnabled` writes Claude-format commands where Codex can use them). For the genuine-CLI provider the same workspace .claude/commands are surfaced in typeahead and typed into the TUI (packages/electron/src/shared/agentWorkflowProviders.ts, NIM-819 comment).

## 2b. Task system: YAML-schema trackers, DB-first items (README's 'plain files' claim is partially historical)
Tracker TYPE schemas are plain YAML on disk per workspace: `{workspace}/.nimbalyst/trackers/*.yaml` (this repo ships task.yaml/bug.yaml/blog.yaml with fields, statuses, sync mode; loaded by `packages/electron/src/main/services/TrackerSchemaService.ts` + `TrackerLoaderService.ts`). Tracker ITEMS are database-first: PGLite `tracker_items` rows with Lexical JSONB content, optionally E2E-encrypted and synced to a TrackerRoom Durable Object (`packages/runtime/src/sync/TrackerSyncEngine.ts`, `TrackerEnvelopeCrypto.ts`; electron `TrackerSyncManager.ts`). Inline `#bug[...]` markers inside markdown docs are indexed as items, and legacy frontmatter plan docs live in gitignored `nimbalyst-local/plans/` (also wired as the SDK plansDirectory in sdkOptionsBuilder.ts:231-233). The migration rationale is design/trackers/unified-tracker-system.md ('Database-first with embedded editors', 43% in-development as of the doc). Agents manipulate tasks via the deferred `nimbalyst-trackers` MCP server (`packages/electron/src/main/mcp/tools/trackerToolHandlers.ts`, tracker_create/tracker_list/tracker_update/tracker_link_session), with prescribed workflows in docs/TRACKER_WORKFLOWS.md; a project can disable AI access entirely.

## 3. Workspace model + where state lives on disk
A workspace/project is a directory opened in an Electron window (path helpers: packages/electron/src/main/utils/workspaceDetection.ts; recent projects in the electron-store). All durable state is in a local DB at `~/Library/Application Support/@nimbalyst/electron/pglite-db/` — currently DUAL-backend (PGLite WASM-Postgres or better-sqlite3, migration in progress; packages/electron/DATABASE.md documents the divergences). Key tables: `ai_sessions`, `ai_agent_messages` (raw transcript, source of truth), `ai_transcript_events` (canonical), `project_state` (per-project window/tabs/layout), `app_settings`, `document_history`, `tracker_items`, plus stores in packages/electron/src/main/services/PGLiteSessionStore.ts / PGLiteAgentMessagesStore.ts / PGLiteWorkspaceRepository.ts (a `workspaces` table) and the workspace-manager window (packages/electron/src/workspace-manager/). Global settings = electron-store (`packages/electron/src/main/utils/store.ts`); per-workspace extras = `.nimbalyst/` (trackers) and gitignored `nimbalyst-local/`. Claude Code's own JSONL session files stay in `~/.claude/projects/` and are imported, so sessions survive outside Nimbalyst. Worktree isolation for parallel sessions: GitWorktreeService.ts/WorktreeStore.ts + docs/WORKTREES.md, session grouping in docs/SESSION_HIERARCHY.md.

## 4. Mobile + collab sync: protocol in-repo, server NOT in-repo, URL only dev-switchable
`packages/collab-protocol/src/` is the single source of truth for the WebSocket wire format: personal.ts (PersonalSessionRoom/PersonalIndexRoom — encrypted session messages, session index, device presence, push tokens, queued prompts, settings sync, and remote-control messages `createSessionRequest`/`createWorktreeRequest`/`sessionControlCommand` that let the iOS app start/drive desktop sessions), teamDocument.ts, teamTracker.ts, teamRoom.ts, projectSync.ts, inbox.ts, roomIds.ts. `packages/collab-adapters/` adapts editor content types for sync. Client engine: `packages/runtime/src/sync/CollabV3Sync.ts` (+ SyncedSessionStore, DocumentSync, TeamSync), orchestrated by `packages/electron/src/main/services/SyncManager.ts`. Everything is E2E encrypted (AES-256-GCM, PBKDF2 key from a passphrase; QR pairing carries server URL + encryption seed — packages/ios/README.md); the server stores opaque blobs. The SERVER (Cloudflare Worker + Durable Objects, 'collabv3') is a separate closed project — root CLAUDE.md references packages/collabv3/ but it is absent from this repo. Repointing: `sessionSync.serverUrl` exists in the settings store (store.ts:147-153) and `packages/electron/src/main/utils/collabSyncUrl.ts` supports `environment:'development'` -> ws://localhost:8790, but ONLY when NODE_ENV!=='production'; packaged builds hardcode wss://sync.nimbalyst.com in SettingsHandlers.ts (~1250-1329), SyncManager.ts:324, ShareHandlers.ts:16, StytchAuthService.ts:1528 and SyncPanel.tsx:126 — self-hosting means patching those constants AND reimplementing the worker against collab-protocol. Disabling is trivial: sync is opt-in (`sessionSync.enabled` false by default, per-project `enabledProjects`), and the app is fully functional offline.

## 5. Auth hooks: full Stytch identity stack already exists; startup is currently un-gated
Identity exists but only gates sync/collab, never app entry. `packages/electron/src/main/services/StytchAuthService.ts` (1,601 lines) implements Stytch B2B: Google OAuth and email magic link, both proxied through the collabv3 worker (desktop never holds the Stytch secret), returning via deep link `nimbalyst://auth/callback?session_token=...&session_jwt=...&org_id=...`; tokens persist in Electron safeStorage; personal-vs-team JWT scoping is enforced with branded types (`packages/runtime/src/auth/jwtScopes.ts`, docs/SYNC_JWT_MODEL.md). Team/org concepts: TeamService.ts, OrgAccessResolver.ts, OrgKeyService.ts, renderer OrgSwitcher.tsx; sign-in UI today lives in Settings -> Sync (packages/electron/src/renderer/components/GlobalSettings/panels/SyncPanel.tsx). The natural insertion point for a mandatory sign-in screen is the existing first-run gate: `packages/electron/src/renderer/components/UnifiedOnboarding/UnifiedOnboarding.tsx` — a modal shown on first launch (role/referral survey + developer-mode choice, driven by `onboarding:get`/complete IPC in packages/electron/src/main/ipc/SettingsHandlers.ts:595) — alongside WelcomeModal/WorkspaceWelcome and the WorkspaceManager picker window (packages/electron/src/renderer/components/WorkspaceManager/WorkspaceManager.tsx). All the hard parts (browser OAuth round-trip, deep-link protocol registration incl. LinuxProtocolRegistration.ts, secure token storage, session refresh) are already built; adding an auth wall is a renderer gating change plus deciding what to do about the documented silent-logout-on-expiry behavior (root CLAUDE.md:43-45).

# RISKS
- The collabv3 sync server is not in the repo (proprietary Cloudflare Worker + Durable Objects); self-hosted sync means reimplementing it against packages/collab-protocol, and production server URLs (wss://sync.nimbalyst.com, collabv3.nimbalyst.workers.dev) are hardcoded in at least 5 files (SettingsHandlers.ts, SyncManager.ts, ShareHandlers.ts, StytchAuthService.ts, SyncPanel.tsx) — the dev override only works in non-production builds.
- Auth (Stytch) is coupled to the same hosted worker as sync — inserting our own sign-in means either standing up a Stytch project + worker equivalent or replacing StytchAuthService's transport, and the personal-vs-team JWT model is documented as the codebase's most-repeated bug source.
- Two database backends (PGLite and better-sqlite3) are simultaneously live mid-migration with divergent JSONB semantics; any fork inherits that complexity and the corruption-prone PID-locked DB files.
- The subscription-billed `claude-code-cli` provider is scaffolding only (sendMessage throws); only the Agent-SDK path is fully functional, and it pins @anthropic-ai/claude-agent-sdk ^0.3.198 plus a CLAUDE_CODE_ENTRYPOINT='cli' classification workaround that Anthropic could invalidate.
- The tracker/task system is mid-migration from plain files to database-first (design doc at 43% progress); building on the file-based model the README advertises would target a layer being deprecated.
- Hard project rule: API keys must never be sourced from process.env (past $100+ billing incident) — any quant-agent preconfiguration must route keys through Nimbalyst settings, not .env files.
